Stage 6에서 PostHog 심고 Stage 7에서 자동 수집한 데이터를 드디어 해석하는 시간

AARRR이 마지막 Stage에 등장하는 이유가 있습니다. Stage 1에서 NSM을 정하고, Stage 6에서 PostHog로 추적을 시작하고, Stage 7에서 주간 리포트를 자동화했습니다. 지표는 이미 쌓여있습니다. 이제 읽는 법을 배우면 됩니다.

🎯 Sean Ellis Test — PMF 측정 골든 룰

"이 서비스가 없어진다면 얼마나 실망하시겠어요?" 한 줄짜리 설문. "매우 실망"이 40% 이상이면 PMF 달성입니다. 50명 이상에게 설문 후 결과로 4가지 시나리오 중 하나에 대응.

SCENARIO 01
PMF 달성 (40%+)
★ 확장 모드
지금부터: 광고 투자 + 채널 확장 + 추천 시스템 강화. 1년 안에 1만 명 목표. 지금이 액셀 밟을 때.
SCENARIO 02
고객층 문제 (특정 그룹만 PMF)
⚠ 페르소나 재설정
지금부터: "매우 실망" 답한 사람 5명 인터뷰 → 공통점 추출 → 새 페르소나로 마케팅 톤 전부 교체.
SCENARIO 03
콘텐츠 문제
⚠ 주제·형식 변경
지금부터: 인기 발행분 vs 비인기 발행분 비교 → 잘 된 것 더 강하게, 안 된 것 폐기. 2개월 후 재측정.
SCENARIO 04
채널 문제 (좋은 사람이 안 들어옴)
⚠ 유입 경로 변경
지금부터: 현재 유입 채널 분석 → ROI 낮은 채널 폐기. 새 채널 1개 집중 (예: 스레드 → 링크드인).

성공 앱 리버스엔지니어링 & 이식

"바닥부터 만들지 말고 성공 앱에서 훔쳐오기" — 합법적 리버스엔지니어링

바이럴로 성공한 뉴스레터(예: 뉴닉·헤드라이트·캐릿)는 "광고 없이 1만 명을 모은 핵심 기능 3가지"가 반드시 있습니다. Antigravity로 해체 분석한 뒤, SDD(Spec-Driven Development)로 내 서비스에 이식.

PHASE 01
🔬 Antigravity로 성공 앱 해체
타겟 앱 1개 선정 → 가입·결제·발송·추천 흐름 모두 직접 체험 → Antigravity로 화면 캡처·HTML 분석.
분석 프롬프트: "이 앱이 광고 없이 1만 명을 모은 핵심 기능 3가지는? 심리적 트리거(FOMO·소속감·성취감)와 기술적 트리거(추천 코드·알림) 분리해서."
PHASE 02
📋 SPEC_TRANSPLANT.md 작성
성공 앱의 핵심 기능을 내 서비스 맥락으로 재설계. 기존 DB 스키마와 충돌 없이 통합.
예: 뉴닉의 "기자 캐릭터화" → 우리 서비스의 "큐레이터 페르소나" / 헤드라이트의 "1줄 요약" → 우리 "5분 안에 따라잡기"로 재해석. 표절이 아닌 추상화 후 재구현.
PHASE 03
🚀 무광고 성장 엔진 장착
이식한 핵심 기능 3가지를 실제 코드로 구현. Claude Code SDD로.
3종 엔진: ① SEO 자동화 — 동적 메타태그 컴포넌트 ② 바이럴 루프 — 추천 코드 시스템(추천 1인당 1개월 무료) ③ 커뮤니티 — 오픈 카카오 채널 또는 스레드 그룹.
⚡ 성공 앱 리버스엔지니어링 프롬프트
한국에서 광고 없이 1만 명 이상 모은 뉴스레터 서비스 3개를 분석해줘. (예: 뉴닉, 헤드라이트, 캐릿) 각 서비스별: 1. 핵심 가치 제안 (한 줄) 2. 광고 없이 성장한 메커니즘 3가지 - 심리적 트리거 (FOMO·소속감·성취감 등) - 기술적 트리거 (추천 코드·알림·바이럴 기능) - 콘텐츠 차별화 (톤·형식·포맷) 3. 가입 → 첫 메일 도착까지의 사용자 여정 (5단계 이내) 4. 유료 전환 후크 (무료 → 유료로 넘어가는 순간의 트리거) 5. 내 서비스(IT/AI/투자 주제, 월 9,900원)에 이식할 만한 패턴 1개씩 (그대로 베끼지 말고 추상화 후 재해석) 마지막에 SPEC_TRANSPLANT.md 작성용 요약: - 이식할 기능 3개 (우선순위 순) - 각 기능의 예상 구현 시간 - 기존 DB 스키마와 충돌 가능성 - 측정할 핵심 지표 마크다운 형식. 분석 근거(어떤 화면·기사·인터뷰)와 함께.
⚡ SPEC_TRANSPLANT.md 작성 프롬프트
위 분석 결과를 기반으로 SPEC_TRANSPLANT.md를 작성해줘. 이식할 기능: [위에서 선택한 1개] 예: 뉴닉의 "큐레이터 페르소나" 기능 구조: [배경] - 어디서 봤는지 (출처) - 왜 효과적인지 (심리·기술적 이유) [현재 상태] - 우리 서비스의 어떤 부분과 연결되나 - 기존 코드 어디를 건드려야 하나 [이식 설계] - 새 컴포넌트 / 새 테이블 / 새 API - 기존과의 통합 지점 - 데이터 흐름 변경 [단계별 구현] - step1.md ~ stepN.md (Stage 5 Harness 형식) - 각 step별 검증 명령 [검증 지표] - 이식 전후 비교할 PostHog 이벤트 - 성공 기준 (예: 7일 내 추천 클릭률 5% 이상) [롤백 계획] - 효과 없으면 어떻게 빼낼지 - Feature Flag로 안전하게 배포 Claude Code Stage 5에서 그대로 실행 가능한 수준으로.

자동화는 검증 후. 5명이 다 "흥미롭지만 안 살 거 같다"면 자동화 만들지 마세요.

안 할 것 리스트 (Won't Do)

"무엇을 안 할지 정한 사람만 무엇을 할지 집중할 수 있습니다"

Stage 2에서 PRD에 Won't Have를 적었다면, Stage 9에서는 사업 차원의 Won't Do입니다. 1년 동안 절대 안 할 것을 명시. 유혹이 와도 흔들리지 않는 헌법.

❌ 첫 1년 절대 안 할 것
유료 광고 (구글·페북·인스타) — SEO·SNS·추천으로 먼저 검증. 광고는 PMF 후.
해외 시장 진출 — 한국에서 1,000명 먼저. 글로벌은 Stage 9의 다음 Stage.
모바일 앱 출시 — 웹 + 이메일로 충분. 앱은 1만 명 이후 검토.
팀 채용 — PMF까지 1인. AI가 팀이고, 채용은 매출 1억 후.
5개 주제 동시 운영 — 1개 주제만 깊게. 다른 주제는 검증 후 추가.
가격 인하·무료 영구 플랜 — 가격은 가치의 신호. 무료는 진정한 고객을 못 거름.
받아본 적 없는 큰 기능 추가 — 사용자 5명이 "이게 있으면 더 좋겠다"고 말한 것만.
경쟁사 따라 만들기 — 경쟁사 분석은 Stage 9 §4에서 끝. 베끼는 게 아니라 추상화.
🎯 Won't Do의 효력
이 리스트는 지인이 "광고 한번 해봐" 권할 때, 친구가 "앱 만들어야지" 말할 때, 본인이 6개월 후 흔들릴 때 꺼내는 헌법입니다. CLAUDE.md와 docs/Roadmap-2026H2.md 둘 다에 박아두세요. 안 할 것이 명확한 사람만 할 것에 집중할 수 있습니다.
